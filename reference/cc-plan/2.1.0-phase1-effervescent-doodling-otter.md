# SherpaNote v2.1.0 Phase 1 - OCR System Upgrade Implementation Plan

## Context

SherpaNote 当前使用单一的 PP-OCR (RapidOCR) 引擎处理所有图像和 PDF 文件。PDF 处理流程为：PyMuPDF 转 PNG 图片 -> RapidOCR 识别。这导致：
- 文本型 PDF 仍走 OCR，效率低、质量差
- 无 Office 文档(DOCX/PPTX/XLSX)直接提取能力
- PyMuPDF (AGPL 许可证) 存在合规风险

本次升级将现有单一引擎重构为**带决策树的智能文档提取系统**，Phase 1 范围：核心重构 + markitdown 集成 + 文本层检测 + 输出标准化 + PyMuPDF 替换。

## Decision Tree

```
[Input File]
  |-- Image (PNG/JPG/BMP/TIFF/WebP) --> PP-OCR (unchanged)
  |-- Office (DOCX/PPTX/XLSX) --> markitdown
  |-- PDF
        |-- has_text_layer? (pdfplumber, first 3 pages, >50 chars)
        |     |-- Yes --> markitdown
        |     +-- No  --> PP-OCR (pypdfium2 PDF->image -> OCR)
        +-- Fallback: PP-OCR (always available)
```

## Risks

1. **pypdfium2 native binary bundling**: PDFium needs to be collected by PyInstaller. Mitigation: test build early.
2. **markitdown dependency tree size**: Pulls pdfplumber, mammoth, python-pptx, openpyxl. Acceptable (~230MB total per PRD).
3. **Segment granularity change**: New markitdown records have line-based segments vs old block-based. Acceptable - frontend uses `transcript` field primarily.
4. **No per-page progress for markitdown**: markitdown doesn't expose progress. Show file-level progress instead.

7. `cd frontend && bun run build` - frontend builds cleanly

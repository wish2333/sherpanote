### 问题1分析+解决

  最可能的原因：huggingface_hub 库在 macOS PyInstaller .app 中首次被调用时，可能触发了 macOS 的某种行为（如 Keychain
  访问、subprocess、安全沙盒等），导致系统重新启动了应用实例（新窗口显示主页面）。

  修复方案：在 Info.plist 中添加 LSMultipleInstancesProhibited，阻止 macOS 创建多个应用实例。

  重新打包 .app 后请测试：（已通过）
  1. 用 HuggingFace 源下载 Whisper 模型 -- 确认新窗口不再出现
  2. 用 HuggingFace 源下载其他模型 (如 Paraformer) -- 确认也没有问题
  3. 如果问题仍然存在，请告诉我，我会进一步排查

  如果添加 LSMultipleInstancesProhibited 后问题仍然存在，说明原因不是多实例问题，而是 pywebview 自身的窗口管理
  bug，需要从 evaluate_js / WKScriptMessageHandler 层面排查。

### 问题2分析+计划+解决

 Plan: AI Processing Result Persistence on Navigation

 Context

 When the user navigates away from EditorView during AI processing (polish, note, mindmap, brainstorm), the results are
  lost. This happens because:

 1. process_text_stream (main.py:419) runs in a background thread and only emits events
 2. Frontend's ai_complete listener (EditorView.vue:284) calls autoSaveResult() to persist
 3. When user navigates away, onBeforeUnmount (EditorView.vue:457) cleans up all listeners
 4. Backend's ai_complete event has no listener -> result never saved to DB

 Note: _auto_process_record (main.py:473) already saves to DB directly. The manual AI processing path should do the
 same.

 Changes

 1. Backend: process_text_stream saves result to DB (main.py:419)

 Add optional record_id parameter to process_text_stream. When provided, the backend persists the AI result to the
 record's ai_results_json field upon completion, before emitting ai_complete.

 - Signature change: process_text_stream(self, text, mode, custom_prompt=None, record_id=None)
 - In _work(): after getting result, if record_id is provided, load the record from storage, merge the AI result, save,
    and emit the updated record data in ai_complete
 - This mirrors how _auto_process_record already works (lines 483-511)

 2. Frontend: Pass record_id to backend (EditorView.vue:324)

 In handleProcessRequest, pass the current record ID to process_text_stream:

 call("process_text_stream", editorText.value, mode, customPrompt ?? null, record.value.id)

 3. Frontend: Update ai_complete handler (EditorView.vue:284)

 When ai_complete includes updated record data from backend, refresh the local record.value to stay in sync:

 offAiComplete = onEvent<{ result: string; truncated?: boolean; record?: TranscriptRecord }>("ai_complete", (detail) =>
  {
   // ... existing logic ...
   // Update local record with backend-saved data
   if (detail.record) record.value = detail.record;
 });

 4. Frontend: Save partial result on unmount (EditorView.vue:457)

 In onBeforeUnmount, if AI processing is still in progress (isAiProcessing.value === true), save whatever accumulated
 text we have so far as a partial result:

 if (isAiProcessing.value && record.value && activeResultMode.value && currentResultContent.value) {
   persistResult(record.value, activeResultMode.value, currentResultContent.value);
 }

 Files to Modify

 - main.py - process_text_stream method (line 419)
 - frontend/src/views/EditorView.vue - handleProcessRequest (line 315), ai_complete handler (line 284), onBeforeUnmount
    (line 457)

  修改总结

  问题

  在 EditorView 中进行 AI 处理（润色/笔记/思维导图/头脑风暴）时，如果用户导航离开，结果会丢失。因为后端只发出事件，由前
  端负责持久化，但前端已卸载了监听器。

  解决方案 — 双重保障

  后端 (main.py):
  - process_text_stream 和 continue_text_stream 新增可选参数 record_id
  - 当提供时，后端在 AI 处理完成后直接保存结果到 SQLite 数据库（通过新的 _persist_ai_result 辅助方法）

## Issue 3: Markdown Display Format Problems

### Root Cause Analysis

**Problem A - Headings have no size difference:**

- Tailwind CSS v4 is used (`@import "tailwindcss"` in `main.css`)
- `MarkdownRenderer.vue` uses `class="prose prose-sm"` which relies on `@tailwindcss/typography` plugin
- **The typography plugin is NOT installed** (`package.json` has no `@tailwindcss/typography`)
- Without the plugin, the `prose` class is a no-op - no heading sizing, no table styles, nothing
- Also, heading regex only matches `#{1,4}`, missing h5/h6

**Problem B - Tables not parsed:**

- The custom markdown parser has zero table support (no `|` parsing logic)
- Even if parsed, without `@tailwindcss/typography`, table CSS wouldn't exist

### Solution

**Option 1 (Recommended): Install `@tailwindcss/typography` + add table parsing**

- Install the plugin: `bun add -D @tailwindcss/typography`
- Import in `main.css`: `@plugin "@tailwindcss/typography"`
- This gives us proper heading sizes, table styles, code block styles, etc. for free
- Only need to add table parsing to the custom renderer
- Minimal changes, uses proven CSS framework

**Option 2: Add all styles manually in CSS**

- Write custom CSS for h1-h6, tables, etc.
- More maintenance burden, reinventing the wheel

## Issue 4: hf-mirror Download Errors

### Root Cause Analysis

**Error 1 - whisper-ggml-large-v3-turbo: SSL timeout to `cas-bridge.xethub.hf.co`**

The download flow:

1. HEAD request to `hf-mirror.com/ggerganov/whisper.cpp/resolve/main/ggml-large-v3-turbo.bin` -> 302 redirect
2. Follow redirect to `cas-bridge.xethub.hf.co/...` (HuggingFace XET storage)
3. SSL handshake to `xethub.hf.co` times out (10s default)

**Root cause**: hf-mirror redirects to the original HF XET infrastructure (`xethub.hf.co`) which is not accessible from China. The hf-mirror only mirrors the API layer, not the XET storage backend. This is a known hf-mirror limitation - it doesn't fully proxy XET-stored files.

This is NOT a code bug - it's an infrastructure limitation. However, we can improve error handling and potentially work around it.

**Error 2 - sherpa-onnx-sense-voice: 404 Not Found**

```
https://hf-mirror.com/csukuangfj/sherpa-onnx-sense-voice-zh-en-ja-ko-yue-int8-2025-09-09/resolve/main/sherpa-onnx-sense-voice-zh-en-ja-ko-yue-int8-2025-09-09.tar.bz2
```

The `download_from_huggingface` function constructs the repo_id as `f"csukuangfj/{model_id}"` when `hf_repo_id` is not set. The sense-voice model has no `hf_repo_id` defined, so it uses `csukuangfj/sherpa-onnx-sense-voice-zh-en-ja-ko-yue-int8-2025-09-09` - but this model may not exist on HuggingFace (the repo might use a different name).

Need to verify: does this file actually exist on huggingface.co? If it doesn't exist on HF either, the model's `sources` should not include `"huggingface"` and `"hf_mirror"`.

### Solution

1. **For the XET timeout issue**:
   - Add retry with increased timeout for HF downloads (default 10s is too short)
   - Consider adding a `HF_HUB_ENABLE_HF_TRANSFER=0` env var to disable the XET transfer path
   - Or pass `timeout=60` to `hf_hub_download()` calls

2. **For the 404 issue**:
   - Verify whether the sense-voice model exists on HuggingFace
   - If it doesn't exist on HF, remove `"huggingface"` and `"hf_mirror"` from its sources
   - Add `hf_repo_id` to models that exist on HF but with a different repo name

3. **General improvement**:
   - Add `timeout` parameter to `hf_hub_download()` calls (default 10s is too aggressive)
   - Set `HF_ENDPOINT` environment variable instead of passing `endpoint` parameter (more reliable)

### Implementation Steps

1. Verify sense-voice model existence on HuggingFace (manual check or via API)

3. In `download_from_huggingface()`: pass `timeout=60` to `hf_hub_download()`
4. In `_install_hf_multi()`: pass `timeout=60` to `hf_hub_download()`
